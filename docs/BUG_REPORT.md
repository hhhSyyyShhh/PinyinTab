## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 项目名称 | PinyinTab 中文路径拼音补全插件 |
| 当前版本 | 0.2.4 |
| 当前平台 | Linux Bash、macOS Zsh |
| 核心语言 | Rust |
| 文档目的 | 记录开发和测试中发现的问题、根因、修复方案、回归测试及隐藏风险 |

本文既记录真正的代码缺陷，也记录容易被误认为 Bug 的使用边界。这样可以避免后续开发者重复排查，也可以作为毕业设计中的“问题分析与系统测试”材料。

## 2. 当前系统工作方式

PinyinTab 不修改真实中文文件名。用户输入拼音并按 `Tab` 后，Shell 插件调用 Rust 核心读取当前目录，把拼音候选转换成真实中文路径，再交回 Bash 或 Zsh。

```text
python3 cfkjb.py + Tab
        ↓
Rust 查找当前目录
        ↓
cfkjb.py 匹配 乘法口诀表.py
        ↓
Shell 将命令行替换为 python3 乘法口诀表.py
        ↓
用户按 Enter，Python 打开真实中文文件
```

因此，目前的关键边界是：必须先按 `Tab` 完成转换，再按 Enter。拼音不是磁盘中真实存在的文件名。

## 4. 已修复问题详述

### BUG-001：新版 Rust 编译失败

#### 现象

```text
expected `fail` to return `()`, but it returns `!`
```

问题代码类似：

```rust
result.unwrap_or_else(fail);
```

#### 根因

`fail` 返回永不返回类型 `!`，而新版 Rust 在该处的类型推导更加严格，要求闭包返回目标类型 `()`。

#### 解决方案

用闭包显式接收错误：

```rust
result.unwrap_or_else(|error| fail(error));
```

#### 验证

`cargo test` 编译成功，Rust 单元测试全部通过。

---

### BUG-002：FUSE 架构与最终需求不一致

#### 现象

第一版实验要求用户从真实目录切换到：

```text
/tmp/pinyintab-mount
```

只有在虚拟挂载目录中，`ceshi.py` 才能访问真实的 `测试.py`。在原始 `demo-source` 中执行会提示文件不存在。

#### 根因

FUSE 提供的是新的文件系统视图，不能让原始目录凭空出现拼音文件名。该架构适合验证“透明别名”，但不符合“不切换目录、仍然进入真实中文目录”的最终需求。

#### 解决方案

取消 FUSE 作为主要交互方式，改为：

```text
Rust 拼音匹配核心 + Bash/Zsh Tab 补全层
```

Shell 在命令执行前把拼音替换成真实中文名称，因此用户始终停留在真实目录。

#### 结论

这是架构层面的需求偏差，不是单个代码错误。FUSE 实验结论保留在文档中，但相关旧源码和挂载脚本已从当前发布包移除，避免用户误用。

### BUG-003：直接输入拼音后回车，程序提示文件不存在

#### 现象

```zsh
python3 cfkjb.py
```

如果未按 `Tab` 就回车，Python 会报告：

```text
can't open file '.../cfkjb.py': No such file or directory
```

#### 根因

当前 PinyinTab 是补全插件，不是内核文件系统，也没有拦截 Enter。磁盘中不存在 `cfkjb.py`，真实文件是 `乘法口诀表.py`。

#### 当前处理

使用流程明确为：

```zsh
python3 cfkjb.py<Tab>
# 变成 python3 乘法口诀表.py
# 再按 Enter
```

#### 状态说明

这属于当前产品边界，不是未修复的代码崩溃。未来可以研究 Zsh `accept-line` Widget，在按 Enter 时尝试转换唯一匹配，但必须防止误改普通命令参数。

### BUG-004：`cat` 错误补全目录

#### 复现

```zsh
mkdir 你好
cat nihao<Tab>
```

旧结果：

```zsh
cat 你好/
```

执行后：

```text
cat: 你好/: Is a directory
```

#### 根因

第一版补全核心没有理解命令语义，对所有命令同时返回文件和目录。

#### 解决方案

新增补全过滤模式：

| 模式 | 典型命令 | 返回内容 |
|---|---|---|
| `--directories` | `cd`、`rmdir` | 只返回目录 |
| `--files` | `cat`、`python3`、`julia`、`javac` | 只返回文件 |
| 默认模式 | `cp`、`mv`、`rm`、`vim` | 文件和目录均可 |
| `--java-classes` | `java` | 只返回去掉 `.class` 的类名 |

同时移除了 Bash 对这些命令的普通文件补全回退，防止 PinyinTab 返回空结果后，Bash 又把目录补回来。

#### 回归测试

测试创建真实目录 `你好/`，确认：

```text
cat nihao<Tab> → 无候选，命令行不变
cd nihao<Tab>  → cd 你好/
```

### BUG-005：Zsh 空结果变成 `/`

#### 现象

即使 Rust 已经正确地为 `cat` 返回空结果，Zsh 仍可能把输入内容替换成 `/`。

#### 根因

旧 Zsh 代码直接把命令输出转换成数组：

```zsh
candidates=("${(@f)$(command)}")
```

当命令没有任何输出时，Zsh 仍生成一个包含空字符串的数组。插件随后把这个“空候选”传给 `compadd -f`，Zsh 将空文件候选解释成当前目录，并显示 `/`。

#### 解决方案

先保存完整输出，再检查是否为空：

```zsh
output="$(command)"
[[ -n "$output" ]] || return 0
candidates=("${(@f)output}")
```

空结果不会再调用 `compadd`。

#### 回归测试

测试脚本模拟 `compadd` 并记录调用参数，确认空结果时调用次数为零。真实 Zsh 中也验证命令行保持 `cat ceshimulu`，不再变成 `cat /`。

### BUG-006：目录斜杠后出现空格

#### 现象

```zsh
cd ce<Tab>
```

补全后得到：

```text
cd 测试目录/ 
```

斜杠后的空格结束了当前参数，用户无法直接继续输入子目录。

#### 根因

Zsh 默认在单个候选后添加空格，即使该候选是目录。

#### 解决方案

把文件和目录候选分组。目录使用：

```zsh
compadd -S ''
```

禁止自动追加空格；普通文件仍保留正常的结尾空格。

#### 说明

目录末尾的 `/` 是正确行为，不应删除。它表示可以继续输入下一层路径。

### BUG-007：多级拼音路径无法解析

#### 现象

真实路径：

```text
测试目录/内部脚本.py
```

旧版只能在父目录已经是中文时尝试匹配：

```zsh
python3 测试目录/neibu<Tab>
```

下面这种纯拼音多层路径无法使用：

```zsh
python3 ceshimulu/neibu<Tab>
```

#### 根因

旧算法直接把 `ceshimulu` 当作真实父目录，用 `read_dir` 打开。磁盘中没有这个拼音目录，所以查找失败。

#### 解决方案

增加逐层路径解析器：

1. 从当前真实目录读取子目录。
2. 把父组件 `ceshimulu` 唯一解析成 `测试目录`。
3. 进入真实的 `测试目录`。
4. 在下一层匹配 `neibu`。
5. 返回完整真实路径。

#### 结果

```text
python3 ceshimulu/neibujiao<Tab>
→ python3 测试目录/内部脚本.py
```

同时支持中文与拼音混合路径。

### BUG-008：末尾斜杠后按 Tab 不能列出下一层

#### 现象

```zsh
cd 测试目录/<Tab>
```

旧算法把末尾 `/` 忽略，错误地再次处理 `测试目录` 本身，而不是读取目录内部。

#### 根因

Rust `Path::file_name` 会忽略路径末尾的分隔符，无法表达“当前叶子名称为空”。

#### 解决方案

不再只依赖 `Path::file_name`，改用最后一个 `/` 手动拆分：

```text
父路径：测试目录
当前输入：空字符串
```

空字符串表示列出父目录中的全部非隐藏候选。

#### 验证

```text
cd ceshimulu/<Tab>
→ cd 测试目录/内部文件夹/

cat ceshimulu/neibuwenjianjia/<Tab>
→ cat 测试目录/内部文件夹/深层说明.txt
```

### BUG-009：缩写主体加扩展名无法匹配

#### 现象

真实文件：

```text
九九乘法表.py
```

拼音索引：

```text
jiujiuchengfabiao.py
```

用户输入 `jiujiu.py` 时，旧版无法匹配。

#### 根因

旧版对完整字符串执行前缀比较：

```text
jiujiuchengfabiao.py starts_with jiujiu.py → false
```

扩展名 `.py` 在用户输入中提前出现，打断了字符串前缀关系。

#### 解决方案

将文件主体和扩展名分开比较：

```text
主体：jiujiu 是 jiujiuchengfabiao 的前缀
扩展名：py 等于 py
```

#### 修复结果

```text
jiujiu.py → 九九乘法表.py
cfkjb.py  → 乘法口诀表.py
cfkjb.jl  → 乘法口诀表.jl
```

### BUG-010：`python3.12` 等命令未稳定适配

#### 现象

`python3` 可以触发文件补全，但用户实际使用：

```zsh
python3.12 cfkjb.py
```

带版本号命令可能使用默认补全器或其他第三方补全器，导致 PinyinTab 没有被调用。

#### 解决方案

显式注册常见版本：

```text
python3.10、python3.11、python3.12、python3.13、python3.14
```

命令类型判断使用 `python[0-9]*`，统一选择文件模式。

#### 验证

真实 macOS Zsh 中验证：

```text
python3.12 cfkjb.py<Tab>
→ python3.12 乘法口诀表.py
→ 正常输出九九乘法口诀表
```

### BUG-012：Java 补全错误地保留 `.class`

#### 现象

```zsh
java 乘法口诀表.class
```

Java 报错：

```text
错误: 找不到或无法加载主类 乘法口诀表.class
```

#### 根因

`java` 命令接收的是类名，不是文件路径。正确命令是：

```zsh
java 乘法口诀表
```

而 `javac` 接收的才是源码路径：

```zsh
javac 乘法口诀表.java
```

通用文件补全无法理解这种差异。

#### 解决方案

为 `java` 增加 `--java-classes` 专用模式：

1. 只读取当前目录中的 `.class` 文件。
2. 用去掉 `.class` 的类名生成拼音。
3. 返回中文类名，但不返回 `.class` 后缀。

#### 结果

```text
javac cfkjb.java<Tab> → javac 乘法口诀表.java
java cfkjb<Tab>       → java 乘法口诀表
```

### BUG-013：Java 内部类干扰主类候选

#### 现象

Java 编译后可能产生：

```text
乘法口诀表.class
乘法口诀表$内部.class
```

内部类通常不能作为用户想直接启动的主类。

#### 解决方案

Java 专用补全过滤包含 `$` 的 `.class` 文件，只保留主类候选。

#### 当前边界

声明了 `package` 的 Java 类必须使用完整类名：

```text
java 包名.乘法口诀表
```

包名与类名不是普通文件路径，尚未实现专用映射。

### BUG-014：测试脚本误删项目示例文件

#### 现象

运行 macOS Zsh 测试后，`demo-source` 中部分示例以及新生成的压缩包消失。

#### 根因

测试脚本使用名称过于通用的全局变量 `fixture` 保存临时目录。Zsh 补全初始化过程中，该全局变量可能被其他函数覆盖。退出时 Trap 执行：

```zsh
rm -rf "$fixture"
```

如果变量已经被覆盖，清理目标就不再是原临时目录。

这是测试基础设施中的严重缺陷，虽然 PinyinTab 核心程序本身没有删除文件的逻辑，但测试脚本同样属于项目代码，必须按严重 Bug 处理。

#### 解决方案

1. 使用项目专用变量名：`PINYINTAB_MACOS_TEST_FIXTURE`。
2. 声明为全局只读变量。
3. Trap 只引用这个只读变量。
4. Bash 测试同步使用 `PINYINTAB_BASH_TEST_FIXTURE`。
5. 测试前后统计项目示例文件和输出文件数量，确认数量不变。

#### 验证

```text
PASS: PinyinTab completion compatibility tests
PASS: PinyinTab macOS/Zsh compatibility tests
demo files: 17 -> 17
output archives/files: 3 -> 3
```

#### 经验

- 临时目录变量必须使用唯一名称。
- 清理路径应该在创建后立即固定，不能依赖可能被覆盖的全局变量。
- 测试脚本中的删除操作也必须接受安全审查。

### BUG-015：关闭插件后原补全规则丢失

#### 现象

如果某个命令原本有自己的 Bash/Zsh 补全器，PinyinTab 开启时覆盖它；旧版 `ptab off` 只删除 PinyinTab 补全，没有恢复原设置。

#### 解决方案

`ptab on` 时保存每个命令原来的补全定义，`ptab off` 时逐个恢复。重复执行 `ptab on` 不会再次覆盖保存值。

#### 回归测试

测试先给 `python3` 注册一个模拟原始补全器，经历 `ptab on` 和 `ptab off` 后比较前后定义，要求完全一致。

### BUG-016：macOS 打包元数据污染

#### 现象

Linux 解压 macOS 创建的包时出现：

```text
tar: Ignoring unknown extended header keyword 'LIBARCHIVE.xattr.com.apple.provenance'
```

还可能出现：

```text
._测试.py
```

#### 根因

macOS 会保存扩展属性和 AppleDouble 元数据，Linux 的 tar 不认识其中部分字段。

#### 解决方案

打包时使用：

```bash
tar --no-xattrs ...
```

并在干净包中排除 `target`、`._*` 等本地构建或元数据文件。

#### 说明

扩展属性警告通常不影响源码，但会干扰用户判断，因此仍应在发布包中消除。

### BUG-017：共同中文前缀后无法继续用拼音筛选

#### 现象

当前目录存在多个同前缀文件：

```text
九九乘法表.py
九九测试程序.py
```

用户输入共同拼音前缀并按第一次 `Tab`：

```zsh
python3.12 jj<Tab>
```

Zsh 发现多个候选，先把共同的真实中文前缀写入命令行：

```zsh
python3.12 九九
```

用户继续输入 `cf`，命令行变成：

```zsh
python3.12 九九cf
```

旧版再次按 `Tab` 时没有结果，无法从模糊候选继续缩小到 `九九乘法表.py`。

#### 根因

旧匹配器只支持三种形式：

1. 真实中文名称前缀。
2. 整个名称的完整拼音前缀。
3. 整个名称的拼音首字母前缀。

`九九cf` 是同一个文件名组件中的“真实中文前缀 + 剩余部分的拼音首字母”。它既不是全中文，也不是整个名称的全拼或首字母，因此三种旧规则全部失败。

#### 解决方案

新增分段混合匹配：

1. 从候选真实名称中寻找已经出现在输入开头的最长中文前缀。
2. 消费这部分真实中文，例如消费 `九九`。
3. 对候选剩余部分重新生成全拼和首字母。
4. 用剩余输入 `cf` 匹配 `乘法表` 的全拼 `chengfabiao` 或首字母 `cfb`。
5. 保留唯一候选 `九九乘法表.py`。

扩展名仍然独立比较，因此下面形式同样支持：

```text
九九cf       → 九九乘法表.py
九九cf.py    → 九九乘法表.py
九九cs       → 九九测试程序.py
```

#### 交互验证

真实 macOS Zsh 测试过程：

```text
python3.12 jj<Tab>
→ python3.12 九九

继续输入 cf，再按 Tab
→ python3.12 九九乘法表.py

按 Enter
→ 正常输出九九乘法口诀表
```

#### 回归测试

Bash 和 Zsh 测试创建多个 `九九` 前缀文件，并验证：

- 初始 `jj` 返回多个候选。
- `九九cs` 只返回 `九九测试程序.py`。
- `九九cheng` 只返回 `九九乘法表.py`。
- `九九cheng.py` 能正确处理扩展名。
- 混合输入可以继续参与下一轮匹配。

#### 当前边界

本次修复针对“已经补出的真实中文前缀 + 后续拼音”场景。它不是通用编辑距离或拼写纠错器，因此拼音漏字、错字、顺序错误仍可能没有结果。

### BUG-018：混合输入仍有多个候选时拼音后缀被删除

#### 现象

当前目录同时存在：

```text
九九乘法表.py
九九除法表.py
```

用户经过第一次补全后输入：

```zsh
python3.12 九九cf
```

其中“乘法”的首字母是 `cf`，“除法”的首字母也是 `cf`，所以两个文件都仍然匹配。旧版按 `Tab` 后，Zsh 删除 `cf`，命令行退回：

```zsh
python3.12 九九
```

再次按 `Tab` 才显示候选，看起来像补全状态在“九九cf”和“九九”之间反复跳转。

#### 根因

Rust 匹配核心正确返回了两个真实候选：

```text
九九乘法表.py
九九除法表.py
```

Zsh 的 `compadd -U` 会计算所有返回候选的共同真实文本前缀。两个候选的共同前缀只有“九九”，因此 Zsh 用“九九”覆盖整个当前输入，连用户已经输入的 `cf` 也一起删除。

这个问题发生在 Shell 展示层，而不是拼音匹配层。

#### 解决方案

当同时满足以下条件时进入“保留并展示”模式：

1. Rust 返回两个或更多候选。
2. 当前输入同时包含中文和英文字母。
3. 说明用户正在共同中文前缀后继续用拼音筛选。

macOS Zsh 不再把真实候选交给替换型 `compadd`，而是使用 `compadd -x` 输出只读候选消息：

```text
PinyinTab 候选（输入仍保留，可继续修改后再按 Tab）：
  九九乘法表.py
  九九除法表.py
```

命令行继续保持：

```zsh
python3.12 九九cf
```

Linux Bash 采用相同判断：把候选打印到终端，并清空 `COMPREPLY`，防止 Readline 用共同中文前缀覆盖混合输入。

#### 精确选择方式

`cf` 本身无法区分“乘法”和“除法”，因为它们的首字母完全相同。用户看到列表后可以修改后半段：

```text
九九cheng<Tab> → 九九乘法表.py
九九chu<Tab>   → 九九除法表.py
```

连续按 `Tab` 会重复显示稳定的候选列表，不再删除 `cf` 或退回“九九”。

#### 真实交互验证

```text
python3.12 jj<Tab>
→ python3.12 九九

输入 cf 并按 Tab
→ 保持 python3.12 九九cf
→ 下方显示乘法表和除法表两个候选

退格删除 cf，输入 cheng 并按 Tab
→ python3.12 九九乘法表.py
```

#### 回归测试

- `九九cf` 的 Rust 结果包含乘法表和除法表。
- Zsh 混合歧义分支只展示候选，不替换当前输入。
- Bash 混合歧义分支清空替换候选，保留 Readline 输入。
- `九九cheng` 唯一匹配乘法表。
- `九九chu` 唯一匹配除法表。
- `九九cs` 唯一匹配测试程序。

#### 当前边界

当前候选列表用于提示和继续输入，尚未实现方向键菜单选择。未来可以增加可选的交互菜单，但不能再次牺牲输入保留行为。

## 5. 容易误判为 Bug 的情况

### 5.1 目录末尾出现 `/`

只要当前命令允许目录，末尾 `/` 是正确结果。它表示候选是目录，并允许继续输入下一层。错误的是把目录提供给只接收普通文件的命令。

### 5.2 Backspace 不自动列出目录

Backspace 的标准行为是删除字符。第二版允许用户删除到 `/` 后按 `Tab` 列出下一层，但不会劫持 Backspace 作为补全键。

### 5.3 拼音拼错

例如：

```text
chengfaoujuebiao.jl
```

比正确的：

```text
chengfakoujuebiao.jl
```

少了 `k`。当前版本不是拼写纠错器，不会自动容忍任意漏字。可以使用更短且准确的首字母：

```text
cfkjb.jl
```

### 5.4 Rust 已安装但找不到 Cargo

Rustup 安装后，当前 Shell 的 `PATH` 可能尚未更新。需要：

```bash
source "$HOME/.cargo/env"
```

这属于 Shell 环境刷新问题，不是 PinyinTab Bug。

### 5.5 Julia 在受限环境中无法创建锁文件

如果 Juliaup 无法写入：

```text
~/.julia/juliaup/.juliaup-lock
```

可能是沙箱或目录权限问题。拼音补全已经完成，但 Julia 运行时自身启动失败，应与路径补全问题分开诊断。

## 6. 已识别但尚未完全解决的隐藏问题

### RISK-001：第三方命令补全器冲突

PinyinTab 开启时会为一组命令注册自己的补全函数。虽然关闭时可以恢复，但开启期间可能失去 Cargo、Git 或某些框架提供的高级补全能力。

建议后续改为“组合补全器”：先运行原补全器，再追加 PinyinTab 候选，或者只在原补全器没有结果时运行 PinyinTab。

### RISK-002：多音字词典不完整

当前只有少量词组覆盖，例如“重庆”“银行”“行长”。真实项目中的“长”“行”“重”“乐”等可能生成错误读音。

建议支持：

- 内置常用词组词典。
- 项目级 `.pinyintab.toml` 自定义读音。
- 用户级多音字覆盖配置。

### RISK-003：拼音重名和中间目录歧义

多个中文名称可能产生相同全拼或首字母。叶子候选可以全部展示，但中间目录必须唯一才能自动进入下一层。

建议未来提供交互选择菜单，并显示真实名称、类型和完整路径。

### RISK-004：直接按 Enter 不转换

用户可能忘记按 `Tab`，直接执行拼音路径。未来可以设计可选的 Enter Widget，但只能在匹配唯一且参数确认是本地路径时转换，否则可能修改程序参数或产生危险操作。

### RISK-005：文件名包含换行符

当前 Rust 与 Shell 使用逐行文本传递候选。如果 Linux 文件名本身包含换行符，会被拆成多个候选。

建议改为 NUL 字节分隔协议，并分别验证 Bash、Zsh 的 NUL 读取方式。

### RISK-006：非 UTF-8 Linux 文件名

Rust 当前只处理能转换成 UTF-8 的名称。无效 UTF-8 文件名会被忽略。中文文件名通常是 UTF-8，但 Linux 并不强制文件名编码。

### RISK-007：Unicode 规范化

macOS 文件系统可能以不同 Unicode 规范化形式保存组合字符。中文汉字通常影响较小，但带音标拉丁字符可能出现视觉相同、字节不同的问题。

建议对输入和索引统一执行 NFC/NFD 规范化测试。

### RISK-008：复杂参数语法

当前主要处理普通本地路径。下面情况需要专用解析：

```text
--file=ceshi.py
scp host:/zhongwen/path
https://example.com/path
find -name '模式'
压缩包内部路径
容器或远程主机内部路径
```

这些参数不能全部当成本地文件路径。

### RISK-009：Java 包名

当前 Java 模式支持当前目录中的无包主类。对于：

```java
package 示例.工具;
```

运行命令必须使用点号分隔的完整类名。未来需要读取 class 元数据或理解源码包声明，不能简单把 `/` 换成 `.`。

### RISK-010：大目录性能

当前每次按 `Tab` 都会调用 Rust 并扫描目录。几百或几千个文件通常可用，但数万个条目的目录可能有明显延迟。

建议增加基准测试、目录缓存、修改时间失效机制，并在测量后再决定是否需要常驻后台进程。

### RISK-011：符号链接和权限

目录解析会跟随可访问的目录符号链接。如果目标无权限、链接失效或循环指向上层，可能返回空结果或错误。虽然当前算法不会主动递归整棵目录树，但仍应增加失效链接和权限拒绝测试。

### RISK-012：危险命令仍然危险

PinyinTab 只负责补全，不会降低 `rm`、`mv` 等命令的风险：

```bash
rm luping<Tab>
```

如果用户确认后执行，删除行为仍由 `rm` 完成。未来可以为破坏性命令增加候选预览，但不应擅自改变原命令语义。

### RISK-013：Shell 配置重载和多次加载

用户可能在 `.zshrc` 中重复 `source` 插件，或者升级后未先关闭旧版本。虽然当前有 active 状态保护，但重新 source 会重新初始化状态变量。

建议安装器增加版本标记，并提供：

```text
ptab status
ptab version
ptab reload
```

### RISK-014：Bash 中使用 `eval` 恢复原补全

Bash 版保存 `complete -p` 的输出，并在关闭时用 `eval` 恢复。内容来自 Bash 自身生成的补全定义，不来自文件名，当前风险可控；但从安全审计角度，未来最好改成结构化保存参数，减少 `eval` 使用。

## 9. 当前结论

PinyinTab 0.2.4 已经解决当前测试中最影响使用的几类问题：

- 不再要求进入 FUSE 虚拟目录。
- 文件命令和目录命令能够区分候选类型。
- Zsh 空结果不再变成 `/`。
- 支持多级纯拼音路径和目录后继续补全。
- 支持缩写主体加扩展名。
- 支持 `python3.12` 等命令。
- Java 运行补全会自动去掉 `.class`。
- 多候选共同中文前缀后可以继续输入拼音二次筛选。
- 混合输入仍然歧义时会保留输入并展示候选，不再删除拼音后缀。
- 测试清理逻辑已经加固。

当前最值得继续研究的是第三方补全器组合、多音字配置、直接 Enter 转换、复杂参数语法和大目录性能。这些问题应该作为后续版本功能，而不是在没有边界和测试的情况下直接加入主流程。
