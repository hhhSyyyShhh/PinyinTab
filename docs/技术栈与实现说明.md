# PinyinTab 技术栈与实现说明

## 1. 项目要解决什么问题

Linux 和 macOS 可以正常保存中文文件名，但在终端里输入中文路径时，用户往往需要频繁切换输入法。PinyinTab 的目标是让用户在真实目录中输入汉语拼音，再通过 `Tab` 把拼音补全成真实中文路径。

例如，磁盘中真实存在：

```text
测试目录/测试.py
```

用户输入：

```bash
cd ceshi<Tab>
python3 ceshi<Tab>
```

补全后，终端里的命令变成：

```bash
cd 测试目录/
python3 测试.py
```

PinyinTab 不修改文件名，不复制文件，也不让用户进入虚拟挂载目录。拼音只负责“输入”，最终执行命令时使用的仍然是真实中文路径。

## 2. 整体架构

```text
用户输入拼音
    │
    ▼
Bash / Zsh 捕获 Tab 补全请求
    │
    ▼
调用 Rust 程序并传入当前目录、当前输入
    │
    ▼
Rust 读取真实目录并生成中文名称的全拼和首字母
    │
    ▼
返回所有匹配的真实中文文件名
    │
    ▼
Shell 展示候选或把拼音替换成真实中文路径
    │
    ▼
Linux / macOS 执行原始命令
```

项目分成两层：

1. Rust 核心程序：负责读取目录、拼音转换、前缀匹配和输出候选。
2. Shell 插件：负责接收 `Tab`、取得当前单词、调用 Rust、把结果交给终端补全系统。

这种分层的好处是：拼音算法只写一次，Bash、Zsh 以及未来的 Fish、PowerShell 都可以复用。

## 3. Rust 语言

### 3.1 为什么使用 Rust

Rust 适合编写这种终端工具：

- 编译成单个本地可执行文件，运行时不要求安装 Python 或 JVM。
- 启动速度快，适合频繁按 `Tab` 时反复调用。
- 内存安全，能减少空指针、越界和释放后使用等问题。
- Linux、macOS 和 Windows 都有成熟的编译目标。
- Cargo 同时负责依赖管理、编译和测试。

Rust 在本项目中不是用来“控制终端界面”的，而是作为跨平台拼音匹配核心。

### 3.2 本项目使用的 Rust 知识

#### Cargo

Cargo 类似 Java 的 Maven/Gradle，也类似 Node.js 的 npm。常用命令：

```bash
cargo test
cargo build --release
```

`cargo test` 编译并运行测试；`cargo build --release` 生成经过优化的正式程序。

#### `Result` 错误处理

读取目录可能失败，例如目录不存在或权限不足。Rust 使用 `Result<T, E>` 表达“成功结果或错误”，调用者必须处理错误，不能无声忽略。

#### `Path` 与 `PathBuf`

路径不能简单地当普通字符串拼接。`Path` 表示借用的路径，`PathBuf` 表示可以拥有和修改的路径。本项目用它们处理当前目录、父目录和多层路径。

#### `std::fs::read_dir`

该接口读取一个真实目录下的条目。PinyinTab 每次补全时读取当前目标目录，然后检查每个文件或子目录是否匹配用户输入。

#### 所有权与借用

这是 Rust 最特殊的基础概念。简单理解：每块数据有明确的所有者；函数临时使用数据时尽量借用，不随意复制。这样可以在编译阶段避免大量内存错误。

## 4. 拼音转换

项目使用 Rust 的 `pinyin` crate。crate 可以理解成别人已经编写并发布的 Rust 功能库。

真实名称 `测试.py` 会生成两个索引：

```text
全拼：ceshi.py
首字母：cs.py
```

匹配采用前缀方式，因此输入 `c`、`ce`、`ceshi` 都可以找到该文件。

中文多音字是一个重要难点。单个汉字脱离上下文时可能有多个读音，例如“行”。当前版本提供少量词组覆盖规则，例如“银行”和“行长”。正式版本需要更大的词典，或者允许用户配置项目级读音。

当多个真实文件产生相同拼音时，PinyinTab 不擅自选择，而是把所有候选交给 Shell 展示。例如两个文件都映射成 `ceshi.py` 时，用户可以从中文候选中选择。

## 5. Bash 补全技术

Bash 提供“可编程补全”机制。关键变量包括：

- `COMP_WORDS`：当前命令被拆分后的单词数组。
- `COMP_CWORD`：光标当前位于哪个单词。
- `COMPREPLY`：补全函数返回的候选数组。
- `complete`：把补全函数注册到某个命令。
- `compopt`：告诉 Bash 按文件名方式处理空格、斜杠等内容。

用户输入 `python3 ce` 并按 `Tab` 时，插件取得当前单词 `ce`，调用：

```bash
ptab complete "$PWD" ce
```

Rust 返回 `测试.py`，插件把它放入 `COMPREPLY`，Bash 再负责替换和显示。

管道不会直接破坏该设计。例如：

```bash
cat 数据.csv | python3 ce<Tab>
```

补全系统处理的是光标当前所在的 `ce`，前面的管道仍由 Bash 自己解析。

## 6. Zsh 与 macOS

现代 macOS 默认使用 Zsh，而不是 Bash。Zsh 的补全体系与 Bash 不同：

- `words`：当前命令的单词数组。
- `PREFIX`：光标前正在输入的当前单词。
- `compadd`：向 Zsh 添加候选。
- `compdef`：把补全函数绑定到命令。
- `compinit`：初始化 Zsh 补全系统。

macOS 版的 Rust 核心与 Linux 版相同，只替换 Shell 适配层。因此不需要 macFUSE，也不需要虚拟目录。

macOS 常见文件系统 APFS 通常不区分英文大小写，而且 Unicode 文件名可能涉及规范化形式。当前中文汉字匹配一般不会受影响，但包含组合音标的拉丁字符需要在正式版增加 Unicode NFC/NFD 归一化测试。

## 7. Shell 与 Rust 之间如何通信

Shell 启动一个很短的 Rust 子进程，并传递：

1. 当前真实目录。
2. 当前输入的路径片段。
3. 是否只返回目录，例如 `cd` 时启用。

Rust 把候选逐行写到标准输出，Shell 读取这些行作为补全候选。这是一种简单的进程间通信方式，不需要网络、数据库或后台服务。

当前逐行协议不支持文件名本身包含换行符的极端情况。未来可以改为 NUL 字节分隔协议。

## 8. 为什么管道和大多数命令可以使用

PinyinTab 并不直接实现 `python3`、`cat`、`cp` 或 `vim`。它只在命令执行前把拼音替换成真实路径。一旦替换完成，后续行为全部由原命令完成。

因此，是否可以使用主要取决于两个条件：

1. 当前参数确实表示本机文件或目录路径。
2. 当前 Shell 在这个位置调用了 PinyinTab 补全函数。

远程路径、URL、Git 引用、`find -name` 模式和程序特有的 `--file=value` 语法不能简单地按普通本地路径处理，需要单独适配。

## 9. 安装与目录结构

安装后主要文件是：

```text
~/.local/bin/ptab
~/.local/share/pinyintab/pinyintab.bash
~/.local/share/pinyintab/pinyintab.zsh
```

`~/.local/bin/ptab` 是 Rust 可执行文件；`pinyintab.bash` 和 `pinyintab.zsh` 分别是两个 Shell 的接入层。

使用 `source` 的原因是：补全函数必须加载进当前 Shell 进程。如果直接启动一个普通子进程，子进程无法修改父终端的补全状态。

## 10. 测试技术

项目包含三类测试：

1. Rust 单元测试：验证中文到全拼、首字母的转换结果。
2. Shell 集成测试：创建临时中文目录，验证空格、嵌套路径、重名、中文输入、英文输入和目录过滤。
3. 交互测试：在真实 Bash/Zsh 中输入命令并按 `Tab`，确认终端最终显示真实中文路径。

测试命令：

```bash
cargo test
./scripts/test-completion.sh
./scripts/test-macos.zsh
```

测试脚本使用临时目录，结束后自动删除，不修改用户项目文件。

## 11. 性能技术点

当前 Demo 每次按 `Tab` 都扫描目标目录。普通项目目录通常足够快，但几万条文件的大目录可能出现延迟。后续可以增加：

- 目录缓存：保存文件名和拼音索引。
- 失效检测：目录修改时间变化后重建缓存。
- 基准测试：测量 100、1,000、10,000 个文件时的耗时。
- 常驻后台进程：减少反复启动 Rust 程序的成本，但会增加系统复杂度。

毕业设计阶段应当先测量性能，再决定是否需要后台服务，不应当仅凭感觉增加架构。

## 12. 安全技术点

- PinyinTab 只读取目录，不主动删除、移动或修改用户文件。
- Shell 候选必须按照文件名规则转义，避免空格被拆成多个参数。
- 开启插件前保存原补全配置，关闭后恢复，减少对用户环境的破坏。
- 不应使用 `eval` 执行来自文件名的内容。
- 安装脚本只写入用户自己的 `~/.local`，不要求 root 权限。

需要注意：如果用户把补全后的中文路径交给 `rm`，删除操作仍然由 `rm` 执行。PinyinTab 不会改变原命令本身的权限和危险性。

## 13. 建议学习顺序

如果此前没有接触过这些技术，可以按下面顺序学习：

1. Linux/macOS 路径、当前目录、相对路径和绝对路径。
2. Bash/Zsh 的变量、函数、数组、`source` 和退出状态码。
3. Rust 基础语法、结构体、`Option`、`Result`、所有权和借用。
4. Cargo 项目结构、依赖与测试。
5. Unicode、UTF-8、中文拼音和多音字问题。
6. Bash programmable completion。
7. Zsh completion system。
8. 集成测试、性能测试和跨平台发布。

不用一开始就把所有知识全部学完。可以先理解“Shell 收集输入—Rust 匹配—Shell 展示候选”这条主线，再逐步深入每一层。

## 14. 后续开发路线

建议按以下顺序继续：

1. 完善 Linux Bash 和 macOS Zsh 的本地路径补全。
2. 增加用户自定义多音字词典。
3. 解决命令自带补全器与 PinyinTab 的组合问题。
4. 支持 `--file=拼音`、引号、波浪号和更多复杂路径语法。
5. 增加目录缓存和性能基准。
6. 使用 CI 自动测试 Linux 与 macOS。
7. 打包为可下载的 GitHub Release，并提供一条命令安装。
8. 最后评估 Fish 和 Windows PowerShell 版本。
